## What went well

- **End-to-end path:** Reciprocal like creates a durable match row; messaging is scoped to match membership.
- **Notifications:** Push wiring respects optional tokens and server-side toggles.
- **Small epic, clear scope:** Easier to keep “done” crisp than multi-track epics.

## Challenges

- **Push reliability:** Provider failures, token rotation—observability (NFR-I2) is ongoing, not one story.
- **Security:** Match access checks must stay centralized—any new message API must reuse `requireParticipant` patterns.

## Insights

- Match + message IT tests become **golden paths** for regressions when Epic 6 adds block enforcement.
- **Celebration modal → chat** handoff needed consistent `otherUserId` for later safety flows—worth explicit mobile types.

## Continuity — Epics 1–2

- Feed and swipe events from Epic 2 are the engine for matches; auth from Epic 1 secures all endpoints.

## Preparation for Epic 4

- Mismatch feedback **does not** require a match—ensure product/UX distinction stays clear in analytics events.
- Server-side reordering must stay **non-explainable** in API—continue opaque feed behavior.
